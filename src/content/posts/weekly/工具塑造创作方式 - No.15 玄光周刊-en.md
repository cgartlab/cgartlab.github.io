## This Week's Topic: Tools Shaping How We Create

Looking back at the creative tools I've used over the past decade, there's a hidden thread that runs through them all: whether hardware or software, tools aren't simply improving efficiency—they're redefining "creation" itself.

The earliest digital creation relied on professional software—Photoshop, Cinema 4D, After Effects, and so on. These programs were closed professional barriers, and the complexity of the tools themselves formed the moat. Entering the mobile internet era, barriers lowered, but the binding relationship between tools and creation didn't change: **you still needed to "learn" the tools to "make" things.** The biggest change in this phase was that tutorials became more abundant, learning became faster, but the barrier remained unmoved.

After 2015, I gradually discovered that tools like Notion, Figma, and Obsidian brought about a subtle but profound shift: these younger, lighter, more advanced tools **began focusing on workflows rather than feature sets**. Creation was no longer a linear process (open software → create new file → start operating), but became modular composition—block editing, bidirectional linking, plugin ecosystems, and template systems became standard. The change in this phase was: **tools evolved from "means of making things" to "collaborative organization of ideas and structures."** And in high-barrier traditional professional workflows, unless tools make revolutionary and disruptive updates, it's hard to shake the process, let alone accelerate collaboration. Traditional cyber craftsmen who want to work faster can only rely on effective communication and practice.

And AI's involvement has pushed this process to a deeper level. Whether in time or money, production costs are declining at a visible rate. One could say that the entire history of tool evolution in the digital age is essentially a history of **continuously declining execution costs**.

From cuneiform script carved on clay tablets, to movable type printing, to Photoshop's layer masks, to Figma's component system, to today where AI can directly generate code—every tool evolution has done the same thing: made the action of "turning ideas into reality" cheaper, faster, and lower-barrier.

When execution becomes cheap, what becomes scarce changes.

The Industrial Revolution made physical labor cheap, so muscles were no longer core competitiveness; the AI era makes cognitive execution cheap, so "knowing how to do" is no longer a barrier. This means that after the execution layer is fully covered by tools, **a creator's irreplaceability can only come from judgment**—the ability to judge what's worth doing amid noise, to judge which direction is meaningful among countless possibilities, and to know what questions you're asking before the tool gives answers.

This isn't a technical question. It's the only question that needs human answering.

More important than embracing tools is being clear about what you're doing with them. More dangerous than rejecting tools is mistaking the tool's judgment for your own. When execution is no longer scarce, those things that are truly scarce begin to surface.

### Jan - Local ChatGPT Alternative

🔗 https://jan.ai

Jan is an open-source local AI application framework that can run a complete ChatGPT-like conversational interface on your computer, supporting connections to Ollama, OpenAI, Anthropic, and other backends. Install and use immediately without any server configuration—data is completely stored locally.

What truly makes Jan noteworthy isn't just the "local operation" feature, but its **extension design**: through the built-in "Hub" system, users can install various pre-configured AI applications—from code review assistants to data analysts, these modules directly reuse your local models without additional fees. For independent developers who value privacy while relying on AI-assisted creation, this provides a possibility for a "self-sufficient" workflow.

When the marginal cost of AI tools approaches zero, localized deployment is no longer just a preference for geeks—it becomes a rational economic choice, and Jan happens to stand at the forefront of this trend.

### Figma MCP Production Pipeline

![Figma dev mode](../_images/15%20玄光周刊-工具塑造的创作方式-1775831292259.webp)

🔗 https://www.francescatabor.com/articles/2026/3/31/building-a-figma-driven-mcp-production-pipeline

Model Context Protocol (MCP) is becoming the bridge between design tools and AI. In March 2026, creative engineer Francesca Tabor detailed the "Figma MCP Production Pipeline" workflow in a practical article: AI can directly read Figma file structure metadata, screenshots, component mappings, and design tokens, then generate production-level UI code—the entire process without human intervention.

This isn't a concept demo, but a workflow that can operate in real projects. AI reads Figma links in PRDs → automatically performs design gap analysis → generates improvement suggestions → produces deployable UI code → finally submits Pull Requests with design compliance audits. The core value of this pipeline is: **it eliminates the most stubborn friction point between design and engineering—"implementation drift from intent."** When design drafts update, code can follow synchronously, rather than going their separate ways after a certain point in time.

The deeper meaning lies in role convergence. The article points out that AI engineers and creative engineers are converging into the same profession—those who can directly translate design intent into production systems. This convergence isn't just about tool upgrades; it means the definition of "design" itself is expanding: design is no longer just output drafts, but structured intent that can drive engineering execution.

This raises a practical question: **Is your design system ready?** The more standardized the components, the more complete the tokens, the more AI can read and reproduce. The quality of design assets directly determines the ceiling of AI output for the first time.

### 2026: AI Singularity Arrives, Reshuffling and Rebirth of Human Society

![via web](../_images/15%20玄光周刊-工具塑造的创作方式-1775831314917.webp)

🔗 https://m.thepaper.cn/newsDetail_forward_32368134

This in-depth report from The Paper systematically reviews the impact of 2026 AI technology development on social structure: structural unemployment will replace large amounts of simple, repetitive white-collar work, AI evolves from assistive tools to active executors, and the social contract itself is being re-examined—when labor is no longer the only prerequisite for survival, the distribution logic of wealth and meaning will fundamentally change.

The core insight of the article isn't technical prediction, but a sociological fact that many overlook: **technological revolution never just changes production tools; it inevitably reshapes production relations.** The steam engine didn't just make factory production faster; it destroyed the guild system of handicrafts and established the wage labor system. AI's impact is equally profound—when "execution" can be completed by machines, "judgment" and "decision-making" become scarce resources, and the gap between those who master these abilities and those who cannot will expand at an exponential rate.

This article raises a question worth seriously facing: In a future where AI can complete most "creative execution," how should human creators position themselves? The answer may not lie in competing with AI on execution efficiency, but in redefining **what must be done by humans**—this question has no technical answer, only a values answer.

### Appwrite - Self-Hosted Backend as a Service

![via Appwrite website](../_images/15%20玄光周刊-工具塑造的创作方式-1775831523848.webp)

🔗 https://appwrite.io

Appwrite is an open-source Backend as a Service (BaaS) platform providing authentication, databases, file storage, cloud functions, real-time subscriptions, and other complete features, supporting one-click Docker deployment. The core difference from Firebase is that from day one it supports fully private deployment—data stays on your own servers.

This makes it an ideal choice for data sovereignty pursuers: enjoying "click-and-use" convenience without handing data over to third parties. The 2026 version further strengthens AI integration—native support for vector databases and embedding APIs, allowing you to build RAG (Retrieval-Augmented Generation) workflows for your own applications without additional external services.

For independent developers, Appwrite provides a pragmatic middle path: no need to build backend infrastructure from scratch, while not giving up control over data. This finds a rare healthy balance between "convenience" and "autonomy."

### TouchDesigner - Real-Time Visual Programming Universe

![via TouchDesigner website](../_images/15%20玄光周刊-工具塑造的创作方式-1775831664666.webp)

🔗 https://zipdo.co/best/visual-programming-software/

TouchDesigner is a real-time visual programming environment developed by Derivative, widely used for stage performance projection, interactive installations, data visualization, and Generative Art creation. It adopts node-based dataflow programming—no need to write a single line of code, by connecting visual operator modules, you can build complex real-time rendering, particle systems, physics simulations, or audio-reactive visuals.

I used TouchDesigner twice while working at a stage lighting art production company. It took some getting used to the node workflow at first, but once accustomed to node-based thinking, you can appreciate what true freedom means—like an absolutely open-world game.

On the other hand, rather than saying TouchDesigner is software, it's more accurate to call it a **creation platform born for live performance**. Its core advantage lies in the real-time feedback of what you see is what you get: every line you connect in the editor projects to the output window the next second. This workflow naturally suits exploratory creation—you can quickly trial and error, layer effects, observe feedback, and iterate on that basis, rather than consuming attention in the loop of writing code, compiling, and running.

For creators, TouchDesigner provides another way to dialogue with tools: not through text (code), but through **visual space itself**. When you organize ideas with nodes rather than statements, creative thinking undergoes subtle changes—you more easily see the global structure of systems, more easily understand how signals flow, how they branch, how they converge. This spatialized thinking habit, in turn, influences other creative fields.

### Skill Integration and Efficiency Revolution

![via web](../_images/15%20玄光周刊-工具塑造的创作方式-1775832027508.webp)

🔗 https://zhuanlan.zhihu.com/p/2024957518648997301

The concept of Skill is quite interesting: it's not a prompt, but a **set of operating manuals containing context, execution steps, and expected results**. Encapsulating know-how into executable modules, allowing Agents to work according to your methods even when you're not present—this is essentially the externalization of experience. But there's a subtle but important trap here: over-encapsulation may cause creators to lose understanding of the process. When you can complete something with a Skill, you should still generally understand how it's done.

For simple tasks, the answer to this question is no. But for complex creation, understanding underlying logic remains indispensable—because true innovation often occurs in the gaps of processes, those places that aren't standardized, that can't be covered by Skills. Skills are accelerators, but they shouldn't become an excuse to bypass learning.

### Agentic Workflow: From Tool to Collaborator

![via web](../_images/15%20玄光周刊-工具塑造的创作方式-1775832585145.webp)

🔗 https://www.tinyash.com/blog/agentic-ai-workflow-tutorial/

This article is quite technical, systematically reviewing the core components of Agentic AI: planner, memory system, tool calling, reflection mechanism, coordinator—five modules form an AI workflow that can autonomously execute multi-step tasks. The author cites a set of data: 76% of enterprises have deployed AI workflows, but over 40% of Agentic AI projects were canceled due to governance issues.

This data deserves more attention than any technical detail. It shows that **the threshold for technical implementation is far lower than the threshold for organizational adaptation**—you can build an AI workflow capable of autonomous execution in a week, but may not solve the question of "who is responsible for AI's decisions" in three months. The essence of governance issues is trust issues: when AI plays an increasingly active role in your workflow, where does your trust in its decisions need to be established?

This question also exists for creators, just on a smaller scale. To what extent are you willing to let AI participate in your creative decisions? Only assisting at the execution level, or can it intervene in judgments of "is this direction right?"

When AI moves from passive recording to active filtering, it's becoming a true participant in the creator's workflow—and the definition of the participant's role needs to be completed by both parties together.
