## This Issue's Theme: Tools Shaping How We Create

Looking back at the creative tools I've used over the past decade or more, there's a hidden thread running through them all: tools don't simply improve efficiency—they're redefining "creation" itself.

Early digital creation depended on professional software—Photoshop, Cinema 4D, After Effects, and the like. These software packages were closed professional barriers, where the complexity of the tools themselves served as the moat. In the mobile internet era, barriers lowered, but the binding relationship between tools and creation didn't change: **you still needed to "learn" the tool before you could "make" something.** The biggest change at this stage was that tutorials became more abundant and learning faster, but the threshold remained.

After 2015, I gradually noticed that tools like Notion, Figma, and Obsidian brought a subtle but profound shift: these newer, lighter, and more advanced tools **began focusing on workflows rather than feature sets.** Creation was no longer a linear process (open software → new file → start operations), but became modular composition—block editing, bidirectional linking, plugin ecosystems, and template systems became standard. At this stage, the change was: **tools evolved from "means of making things" to "collaborative organization of ideas and structure."** In traditional high-barrier professional workflows, tools couldn't disrupt the process without revolutionary updates, let alone accelerate collaboration. Traditional cyber-craftspeople who wanted to work quickly could only rely on effective communication and practiced hands.

AI's intervention pushed this process to a deeper level. Production costs—both in time and money—are declining at visible speeds. We can say the entire history of digital-era tool evolution is essentially **the history of continuously declining execution costs.**

From cuneiform on clay tablets, to movable type printing, to Photoshop layers and masks, to Figma's component systems, to AI generating code directly—every tool evolution has done the same thing: making "turning ideas into reality" cheaper, faster, and lower barrier.

When execution becomes cheap, what becomes scarce changes.

The Industrial Revolution made physical labor cheap, so muscles were no longer a core competency. The AI era makes cognitive execution cheap, so "knowing how to do things" is no longer a barrier. This means that after tools comprehensively cover the execution layer, **a creator's irreplaceability can only come from judgment**—judging what's worth doing amid noise, determining which direction is meaningful among countless possibilities, knowing what question you're asking before the tool gives you an answer.

This isn't a technical question. It's the only question that requires a human answer.

What's more important than embracing tools is being clear about what you're doing with them. What's more dangerous than rejecting tools is mistaking the tool's judgment for your own. When execution is no longer scarce, those things that truly are scarce begin to surface.

### Jan - A Locally-Running ChatGPT Alternative

🔗 https://jan.ai

Jan is an open-source local AI application framework that runs a complete ChatGPT-like conversational interface on your computer, supporting connections to Ollama, OpenAI, Anthropic, and various other backends. Install and use—no server configuration needed, with all data stored locally.

What truly makes Jan worth attention isn't just the "local operation" feature, but its **extensible design**: through the built-in "Hub" system, users can install various pre-configured AI applications—from code review assistants to data analysts—these modules directly reuse your local models without additional costs. For independent developers who value privacy while relying on AI-assisted creation, this offers a "self-sufficient" workflow possibility.

When the marginal cost of AI tools approaches zero, local deployment isn't just a geek preference—it becomes a rational economic choice, and Jan happens to stand at the forefront of this trend.

### Figma MCP Production Pipeline

![Figma's Dev Mode](../_images/15%20玄光周刊-工具塑造的创作方式-1775831292259.webp)

🔗 https://www.francescatabor.com/articles/2026/3/31/building-a-figma-driven-mcp-production-pipeline

The Model Context Protocol (MCP) is becoming a bridge between design tools and AI. In March 2026, creative engineer Francesca Tabor detailed the "Figma MCP Production Pipeline" workflow in a hands-on article: AI can directly read Figma file structural metadata, screenshots, component mappings, and design tokens, then generate production-level UI code—with no manual intervention required.

This isn't a concept demo; it's a workflow that can operate in real projects. AI reads Figma links from PRD → automatically performs design gap analysis → generates improvement suggestions → outputs deployable UI code → finally submits a Pull Request with design compliance audit. The core value of this pipeline lies in: **it eliminates the most stubborn friction point between design and engineering—"drift between implementation and intent."** When design files update, code can follow synchronously, rather than going their separate ways after a certain point in time.

The deeper significance lies in role fusion. The article points out that AI engineers and creative engineers are converging into the same role—those who can directly translate design intent into production systems. This fusion isn't simply a tool upgrade; it means the definition of "design" itself is expanding: design is no longer just deliverables, but structured intent that can drive engineering execution.

This raises a pragmatic question: **Is your design system ready?** The more standardized your components and the more complete your tokens, the more AI can read and reproduce. For the first time, the quality of design assets directly determines the upper limit of AI output.

### 2026: AI Singularity Arrives, Restructuring and Renewal of Human Society

![Via Web](../_images/15%20玄光周刊-工具塑造的创作方式-1775831314917.webp)

🔗 https://m.thepaper.cn/newsDetail_forward_32368134

This in-depth report from Pengpai News systematically reviews how AI technology development in 2026 affects social structure: structural unemployment will replace large amounts of simple, repetitive white-collar work; AI evolves from auxiliary tool to active executor; and the social contract itself is being re-examined—when labor is no longer the sole prerequisite for survival, the logic of wealth and meaning distribution will fundamentally change.

The article's core insight isn't a technical prediction, but a sociological fact overlooked by many: **technological revolution never just changes production tools—it inevitably reshapes production relations.** The steam engine didn't just make factory production faster—it destroyed the guild system of handcraft, establishing the wage labor system. AI's impact is equally profound—when "execution" can be done by machines, "judgment" and "decision-making" become scarce resources, and the gap between those who master these abilities and those who cannot will expand at an exponential rate.

This article raises a question worth serious consideration: In a future where AI can handle most "creative execution," how should human creators position their role? The answer may not lie in competing with AI on execution efficiency, but in redefining **what must be completed by humans**—this question has no technical answer, only a values-based one.

### Appwrite - Self-Hosted Backend-as-a-Service

![Via Appwrite Official Site](../_images/15%20玄光周刊-工具塑造的创作方式-1775831523848.webp)

🔗 https://appwrite.io

Appwrite is an open-source Backend-as-a-Service (BaaS) platform providing complete features including authentication, database, file storage, cloud functions, and real-time subscriptions, with one-click Docker deployment. Its core difference from Firebase: it supports fully private deployment from day one, with data staying on your own servers.

This makes it the ideal choice for data sovereignty pursuers: enjoying "click-and-use" convenience without handing data over to third parties. The 2026 version further strengthens AI integration—native support for vector databases and embedding APIs, allowing you to build RAG (Retrieval Augmented Generation) workflows directly for your applications without additional external services.

For independent developers, Appwrite offers a pragmatic middle path: no need to build backend infrastructure from scratch, while not giving up control over data. It finds a rare, healthy balance between "convenience" and "autonomy."

### TouchDesigner - A Creative Universe of Real-Time Visual Programming

![Via TouchDesigner Official Site](../_images/15%20玄光周刊-工具塑造的创作方式-1775831664666.webp)

🔗 https://zipdo.co/best/visual-programming-software/

TouchDesigner is a real-time visual programming environment developed by Derivative, widely used in stage performance projections, interactive installations, data visualization, and Generative Art creation. It employs node-based dataflow programming—no code needed; by connecting visual Operator modules, you can build complex real-time rendering, particle systems, physics simulations, or audio-reactive visuals.

I used TouchDesigner twice during my time at a stage lighting and art production company. Getting started, the node workflow wasn't intuitive, but once I adapted to node-based thinking, I discovered what true freedom really means—it's like an absolutely open-world game.

On the other hand, rather than saying TouchDesigner is software, it's more accurate to call it **a creation platform built for live performance.** Its core advantage lies in real-time feedback: every line you connect in the editor appears in the output window the next second. This working method is naturally suited for exploratory creation—you can quickly experiment, layer effects, observe feedback, then iterate, rather than draining attention in a code-compile-run loop.

For creators, TouchDesigner offers another way to dialogue with tools: not through text (code), but through **visual space itself.** When you organize ideas with nodes instead of statements, creative thinking shifts subtly—you more easily see the system's overall structure, understand how signals flow, branch, and converge. This spatial thinking habit conversely influences other creative domains.

### Skill Integration and the Efficiency Revolution

![Via Web](../_images/15%20玄光周刊-工具塑造的创作方式-1775832027508.webp)

🔗 https://zhuanlan.zhihu.com/p/2024957518648997301

The concept of Skill is intriguing: it's not a prompt, but a set of **operation manuals containing context, execution steps, and expected results.** Packaging know-how into executable modules allows Agents to work your way even when you're not present—this is essentially the externalization of experience. But there's a subtle yet important trap here: over-encapsulation may cause creators to lose understanding of the process itself. When you can accomplish something with a Skill, you'd better still have a general sense of how it's done.

For simple tasks, the answer is no—you don't need to. But for complex creation, understanding underlying logic remains indispensable—because true innovation often happens in the gaps between processes, in those places that aren't standardized or covered by Skills. Skill is an accelerator, but it shouldn't become an excuse to skip learning.

### Agentic Workflows: From Tools to Collaborators

![Via Web](../_images/15%20玄光周刊-工具塑造的创作方式-1775832585145.webp)

🔗 https://www.tinyash.com/blog/agentic-ai-workflow-tutorial/

This article is quite technical, systematically sorting through the core components of Agentic AI: planners, memory systems, tool calling, reflection mechanisms, orchestrators—five modules forming an AI workflow capable of autonomously executing multi-step tasks. The author cites data: 76% of enterprises have deployed AI workflows, but over 40% of Agentic AI projects have been cancelled due to governance issues.

This data deserves more attention than any technical detail. It shows **the barrier to technical implementation is far lower than the barrier to organizational adaptation**—you might build a self-executing AI workflow in a week, but spend three months unable to solve "who is responsible for AI's decisions." The essence of governance problems is trust: when AI plays an increasingly proactive role in your workflow, where does trust in its decisions need to be established?

For creators, this question exists at a smaller scale. How much are you willing to let AI participate in your creative decisions? Only as execution-level assistance, or can it介入到 "is this direction correct" judgments?

When AI evolves from passive recording to active filtering, it becomes a true participant in the creator's workflow—and the definition of a participant's role needs to be jointly established.
